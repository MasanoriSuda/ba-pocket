# Design (0002 Define PoC Personas)

## 1. 実装アプローチ
- ドキュメントのみで完結する（コード変更なし）
- PoC制約に沿って「入力負担」「安全性」「2週間で観測可能な行動」を軸にペルソナを設計する
- 各ペルソナは同一フォーマットで記述し、比較可能性を担保する

## 2. 変更する成果物
- `.steering/0002-define-poc-personas/requirements.md`（ペルソナ定義）
- `.steering/0002-define-poc-personas/design.md`（設計）
- `.steering/0002-define-poc-personas/tasklist.md`（作業タスク）

## 3. ペルソナ設計方針
### 3.1 価値検証の軸
- 「今夜の指示が短く明確で、安心して実行できる」価値が伝わる像に限定

### 3.2 入力負担の許容範囲
- 問診は2〜3問まで、手持ち登録は3〜12個程度まで許容できる像を想定

### 3.3 安全性の境界
- 医療行為・診断期待を持つ層はネガティブにまとめ、受診誘導が刺さる像にする

### 3.4 観測可能性（2週間）
- 再訪、提案実行、翌日フォローの行動が想定できる状況設定を入れる

## 4. ペルソナ記述テンプレート
- 基本属性: 年代/生活/スキンケア経験
- 肌状態: 普段/荒れる時/頻度
- 荒れるトリガー: 上位3つ
- 家にあるスキンケア量: 少/中/多（カテゴリ内訳）
- 困りごと（1文）
- このアプリに期待する「今夜の答え」（1文）
- 入力許容度: 問診は何問までOK？ 手持ち登録は何個までOK？
- 成功体験（役に立った瞬間）
- 失敗条件（使わなくなる理由）
- 懸念/リスク（安全・医療誤解の観点）
- PoCで観測する主要指標

## 5. 利用シナリオ設計
- トリガー → Home導線 → 問診回答の特徴 → 手持ち入力の特徴 → 結果画面で刺さる表現 → 翌日フォローの反応
- 1ペルソナにつき1本、短文で記載する

## 6. スコープ外の明文化
- N1に紐づけて対象外症状を具体的に列挙する
- 対象外時のアプリ挙動（受診誘導、注意事項、攻め成分中止）を固定で記載する

## 7. リスクと回避策
- リスク: 医療的な誤解が生じる → 回避: N1で期待値調整と受診誘導を明示
- リスク: 入力負担が重く見える → 回避: 入力許容度を数値で記述
- リスク: 価値検証が曖昧になる → 回避: KPIをペルソナ別に紐づける
